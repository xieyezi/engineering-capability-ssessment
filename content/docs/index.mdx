## 一、对 TypeScript 类型系统理解不足

**现象：** 7 个核心文件使用 `: any` 声明类型（包括 `zod-validation-utils.ts`
这类底层工具库），37+ 个文件使用 `as any` / `as unknown as` 绕过类型检查。

这说明几个层次的问题：

1. **不理解自己所写的函数签名**——在核心校验工具中用
   `: any`，说明他写函数时不清楚输入输出该是什么类型
2. **遇到类型报错选择绕过而非根治**——`as any`
   是 TypeScript 的「逃生舱」，偶尔使用可以理解，但 37+ 个文件遍地开花就是习惯性问题
3. **无法正确对接生成的 API 类型**——Orval 自动生成的类型与他的代码大量不匹配，说明他读不懂生成的类型定义，或是写的代码没有遵循 API 契约

一个具备基本 TypeScript 工程能力的开发者，应该能用正确的类型守卫、泛型约束、类型收窄来解决问题，而不是到处
`as any`。

## 二、不会管理复杂度

**现象：** Demat 请求创建向导是一个多步骤表单（SetupStep → RegisteredHolderDetails →
CertificateSection → DestinationStep → DocumentAndPaymentStep →
ReviewStep，共 6 步）。当需求变更需要对数据结构做一个微调时——例如在请求中新增一个可选字段——每一个步骤组件都需要逐个打开修改 props 定义、调整传参、更新渲染逻辑。

**具体来说，这意味着什么：**

假设后端在 Demat 请求中新增了一个
`preferredSettlementDate`（首选结算日期）字段。在一个设计合理的系统中，你只需要：

1. 在 schema / types 中添加字段定义
2. 在对应的那一个步骤组件中添加日期选择器

但在 John 的实现中，你需要修改 **6 个文件**：

```
config.ts              ← 更新表单字段定义
SetupStep.tsx          ← 修改组件 props 类型，传递新字段
RegisteredHolderDetails.tsx ← 修改组件 props 类型，继续向下传
CertificateSection.tsx ← 同上
DestinationStep.tsx    ← 同上
DocumentAndPaymentStep.tsx ← 同上
ReviewStep.tsx         ← 修改展示逻辑，新增一行展示新字段的值
```

**根本原因：**
数据没有通过一个统一的 store 或 context 管理，而是作为 props 从顶层组件逐层手动传递到每一个子组件。这使得任意一个数据结构变更都会沿组件树向下「泄漏」，形成**一改全改**的局面。他没有在组件之间建立隔离层，也没有定义稳定的接口契约。这说明他缺乏设计组件通信边界的能力——分不清哪些实现细节应该封装在组件内部，哪些数据应该在组件间共享。

## 四、不理解分页处理

**核心问题：**
他没有理解分页的本质——数据应按需、按页加载，而非一次性全量获取后在前端假分页。他在代码中将 `size`
默认值设为 `100000`，以此绕过真正的服务端分页。

### 证据

| 文件                                                                 | 代码                                | 问题                              |
| -------------------------------------------------------------------- | ----------------------------------- | --------------------------------- |
| `src/hooks/use-demat-requests.ts#L32`                                | `'/api/demat-requests?size=100000'` | 硬编码在 URL 中，完全没有分页参数 |
| `src/hooks/use-demat-summary.ts#L12`                                 | `'/api/demat-requests?size=100000'` | 获取全量数据仅为了在前端统计数量  |
| `src/hooks/use-user-registrations.ts#L35`                            | `size = 100000`                     | 默认值即全量，分页形同虚设        |
| `src/hooks/use-usi-application.ts#L30`                              | `size = 100000`                     | 同上                              |
| `src/hooks/use-shareholders.ts#L66`                                  | `size = 100000`                     | 同上                              |

### 这说明什么

1. **不理解分页的意义**——分页的目的是减少单次数据传输量、降低后端压力、提升首屏响应速度。`size=100000`
   完全违背了这些目标。他可能认为分页只是一个「参数」，而非一种数据加载策略。

2. **用假分页掩盖真问题**——这些 hooks 的接口上都有 `page` 和 `size` 参数（看起来支持分页），但
   `size` 的默认值直接设置为 `100000`，意味着只要调用方不显式传入较小的
   `size`，数据就会一次性全部拉回。这是典型的假分页——接口设计看起来正确，实际行为却是全量获取。

3. **最离谱的例子：`use-demat-summary`**——这个 hook 的目的仅仅是统计各来源的请求数量，却通过
   `size=100000` 把全量 Demat 请求拉到前端再逐一计数。正确的做法应该是后端提供一个
   `/api/demat-requests/summary`
   接口，或至少在查询参数中指定只要聚合数据。这说明他**完全没有「计算应该发生在服务端」的意识**。

## 五、缺乏表单设计能力，交付物达不到企业级 SaaS 标准

**核心问题：**
他交付的表单页面在视觉布局、交互体验、组件选型等方面均未达到企业级 SaaS 产品的基本门槛，几乎每一个表单页面在交付后都需要他人返工重做。

这不是"做得不够好"的问题，而是**从第一版就不具备上线标准**——不是审美层面的差距，而是工程层面的缺失：没有信息架构、没有交互考量、没有组件选型判断。

---

### 案例分析：User Registration 表单

以 `src/app/intranet/operations/user-registrations/register/page.tsx`
为例，该表单包含约 20 个字段，拆分在 4 个 Card 中，但存在以下问题：

#### 1. 字段没有分类和优先级

表单现有 4 个 Card：`EmailAddressCard`、`PersonalIdInformationCard`、`CommunicationInformationCard`，外加一个外包的提交 Card。

但 `PersonalIdInformationCard` 内部混杂了三类完全不相关的信息：

- **身份信息**：姓名（中/英文）、证件类型、证件号码、证件有效期
- **协议确认**：4 个独立的 checkbox（年龄确认、信息准确、同意条款、隐私政策）

协议确认本质上是法律合规信息，与个人身份信息在语义上毫无关联，却被塞进同一个 Card。这说明他**没有对字段做语义分类**——只是把"剩下的"字段丢到了一个 Card 里。

#### 2. 缺乏视觉层级和优先级

表单中所有 `<Input>` 组件完全相同，没有区分核心字段和次要字段：

```tsx
// 核心字段：姓名
<Input id="first-name" {...field} />

// 次要字段：地址行 4（同等级别展示）
<Input id="address-line-4" {...field} />
```

在金融表单中，姓名、证件类型、证件号码是用户必须首先关注的**核心信息**，而地址行 3、行 4 是**补充信息**。但它们被赋予了完全相同的视觉权重——相同的宽度、相同的高度、相邻排列。

#### 3. 全宽布局导致视觉空旷

表单外层包裹结构是 `mx-4 lg:mx-6`，内部使用
`grid grid-cols-1 md:grid-cols-4`。在宽屏下，4 列布局将所有 `<Input>`
拉伸到等宽，产生大量空白区域，用户体验松散。

#### 4. 组件选型不当

| 场景         | 实际使用                         | 问题                                                                     |
| ------------ | -------------------------------- | ------------------------------------------------------------------------ |
| 证件有效期   | `<Input type="date">`            | 原生日期控件在不同浏览器体验不一致，金融系统应使用 DatePicker            |
| 4 个协议确认 | 4 个独立的 `<Checkbox>` 竖排堆叠 | 没有分组容器、没有全选/批量操作，视觉冗长                                |
| 地址行 1-4   | 4 个等权 `<Input>`               | 地址行 1 是主地址，应与行 2-4 有视觉区分；超过 2 行时应考虑 `<Textarea>` |
| 邮件地址     | 独立一个 Card                    | 一个字段占整个 Card，过度包装                                            |

---

### 后果：几乎都需要返工

以上问题不是边缘情况，而是**系统性的**——他交付的每个表单页面都呈现出同样的模式。这意味着：

1. **他交付的页面达不到上线标准**，必须由其他人介入修改布局、调整组件、重新分组字段
2. **返工成本被转嫁给团队**——Review 时发现问题 → 退回修改 → 他改不明白 → 别人接手重做
3. **在金融 SaaS 场景下尤其致命**——客户对表单的专业感、信息密度、操作效率有极高要求，一个粗糙的表单会直接拉低整个产品的可信度

---

### 这说明什么

1. **不会做字段分析**——拿到表单需求后，他没有先对字段进行分类（核心 vs 补充、必填 vs 可选、个人信息 vs 合规信息），而是直接平铺或随机分组。

2. **不具备表单布局意识**——他不知道表单宽度应该有限制（通常 600-720px 为舒适阅读宽度）、不知道两栏/三栏布局的适用场景、不知道输入框应该根据预期内容长度调整宽度（姓名短、地址长）。

3. **不会选择 UI 组件**——面对"选日期"需求，他用原生 `<input type="date">`
   而不是 DatePicker；面对"长文本地址"，他用 `<Input>` 而不是
   `<Textarea>`。他缺少对组件库功能的了解，也没有"这个场景适合用什么组件"的判断力。

4. **缺乏企业级产品的质量标准认知**——他可能从未见过或研究过一个成熟 SaaS 产品的表单应该长什么样，因此交付出来的东西停留在「能跑就行」的层面，对于「好不好用」「专不专业」没有概念。

## 六、典型案例：`MemorizedDataTable` 组件设计分析

`MemorizedDataTable` 是 John 开发的一个通用表格组件（提交
`3cfdd35`），以下是该组件暴露出的具体设计问题：

### 1. 数据获取与 UI 层耦合

该组件在内部通过 `fetchHook`
自行执行数据请求，导致其同时承担了「数据容器」和「展示组件」两个角色。这违背了 React 社区广泛认可的
`UI = f(state)` 原则——UI 组件应当只消费状态并专注渲染，数据获取和编排应留在页面/容器层。

**具体问题：**
当页面需要统一管理缓存策略、重试逻辑或请求去重时，部分数据获取行为已经嵌入在表格组件内部，无法从页面层集中控制。

**正确做法：** 数据获取应上移到页面级的 TanStack Query（`useQuery`），`MemorizedDataTable`
退化为纯展示组件。

---

### 2. 存在重复请求风险

当页面同时需要表格数据和基于同一数据源派生的其他信息时，页面和表格会各自独立触发请求，产生重复的网络调用。

**具体问题：**
摘要卡片和表格列表背后是同一个接口，但因为页面和表格各自管理自己的数据获取，同一接口被调用了两次。

这说明他在设计组件接口时**没有考虑数据共享和请求去重**的场景——本质上是缺乏对组件在整体数据流中位置的思考。

---

### 3. 轮询行为被硬编码为组件契约

`fetchHook` API 与 `intervalMs`（默认 5 秒）强绑定，将轮询语义强行植入了一个通用表格组件。

**具体问题：**
低频参考数据页面只需要初始化时加载一次，却被迫适配轮询导向的 API 模型。这属于**将特定业务场景的需求固化为通用组件的行为**——缺乏对组件泛用性的考量。

---

### 4. 暴露了未实现的 API

`onRowSelect` 等 props 已在组件接口中公开声明，但实际上并未接入组件内部逻辑，回调永远不会被触发。

**具体问题：** 调用方预期 `onRowSelect`
能驱动详情面板更新，结果发现是个空壳。这表明他在**设计 API 时没有做到「声明即实现」**——可能是先定义了接口打算后续实现，但最终遗留了半成品代码。

---

### 5. 行标识契约隐式依赖 `row.id`

组件内部隐式依赖 `row.id`
作为选择功能和表格标识的依据，但实际后端数据可能使用不同的唯一性规则，部分数据集甚至根本没有 `id`
字段。

**具体问题：** 当页面需要用 `accountNo + market`
作为联合主键来标识记录时，组件无法适配，可能导致选择行为异常。

他没有意识到**行标识规则是业务概念而非组件假设**，应当暴露显式的 `getRowId` 回调让调用方定义：

```tsx
getRowId={(row) => `${row.accountNo}-${row.market}`}
```

---

### 该案例映射的能力短板

以上 5 个问题并非孤立存在，它们精确指向了前述评估中的关键缺陷：

| 案例问题          | 映射能力短板                      |
| ----------------- | --------------------------------- |
| fetchHook 耦合 UI | 复杂度管理、组件边界设计          |
| 重复请求          | 数据流建模、复杂度管理            |
| 轮询硬编码        | API 设计、泛用性考量              |
| onRowSelect 空壳  | 半成品交付、接口契约设计          |
| 隐式依赖 row.id   | 接口契约设计、TypeScript 类型建模 |

---

### 这说明什么

这不是"选型多样化"，而是**没有选型**——他没有在引入依赖之前做任何评估和取舍。具体来说：

1. **没有调研就引入**——看到某个博客/教程用了什么库，就直接
   `npm install`，没有比较同类方案，也没有考虑这个库是否与项目中已有的方案冲突。

2. **不懂"少即是多"**——有经验的工程师在选择依赖时是保守的：每引入一个包都是一项长期维护承诺。他缺乏这种意识，把
   `package.json` 当收藏夹用。

3. **不懂收敛技术栈**——他没有"团队应该用一种方式写样式""全局只用一个表格库"的工程纪律。结果就是团队成员在不同页面用不同方案，代码无法互相借鉴，也无法建立最佳实践。

---

### 实际后果

1. **安全面 ×58**：npm 依赖包数量暴增，每个包都是独立攻击面。出一个 CVE，修一个，无法批量处理。

2. **升级锁死**：MUI v5 → v6 要改 API，ag-grid v30 →
   v31 要改 API，每多一个库就多一条升级阻断链。5 套 CSS 方案中任何一套的大版本升级都可能 break 现有页面。

3. **新人无从下手**：没有一个文档告诉你"这个项目用哪个表格库"，只能翻历史代码猜测——而不同页面答案不同。

4. **重构极其困难**：假设团队决定把样式统一到 Tailwind、表格统一到 react-table——这意味着要把 A 页面的 Emotion 改写成 Tailwind、B 页面的 styled-components 也改写成 Tailwind、C 页面的 Sass 也改写成 Tailwind，且每个页面要保证视觉效果不退化。同时表格要从 ag-grid
   / react-virtualized / react-window /
   react-virtuoso 逐一迁移到 react-table，每个表格的功能（排序、筛选、虚拟滚动、行选择）都要用新 API 重新实现。这不是改几个文件的问题，而是**需要同时理解 5 套方案才能安全拆掉其中 4 套**——而这种跨方案的理解力要求，恰恰是当初随意引入这些依赖时所没有考虑到的代价。

---

## 总结

| 能力维度           | 核心表现                                   |
| ------------------ | ------------------------------------------ |
| TypeScript         | 大量 `: any` 和类型断言绕过                |
| 复杂度管理         | 组件紧耦合，一改全改                       |
| 模块分类与领域建模 | 关联模块平铺，缺少层级分组                 |
| 分页处理           | size=100000 绕过服务端分页，全量拉取       |
| 表单设计           | 字段无分类优先级，组件选型不当，布局空旷   |
| 沟通响应           | Issue 分配后 4-5 天甚至一周不处理          |
| 技术选型           | 5 套 CSS + 32 个 UI 包，功能重叠无统一标准 |
